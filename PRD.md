## 一、产品概述

**Peemail** 是一款基于地理位置的社交应用，灵感来自狗狗用气味标记领地——你路过某个地方，留下一点什么，陌生人嗅到了，世界悄悄连接。

> 发帖 = **留个味儿** | 看帖 = **嗅一嗅** | 浏览计数 = **闻过**（自动） | 表态 = **气味相投 / 臭味相投**（主动，同意 / 不同意）

### 核心差异点

| 维度 | Peemail | 传统社交 |
|------|---------|---------|
| 连接触发 | 地理位置 | 主动加好友 |
| 内容时效 | 公开气味钉有时效，私密日记永久 | 永久保留 |
| 身份系统 | 无需注册，设备绑定 | 账号注册 |
| 场景 | 探索城市、通勤 | 维护关系链 |

## 二、核心功能与边界

### 2.1 功能清单

#### 模块 A：地图留言板（Map Board）— **MVP 第一阶段唯一重点**

**核心功能**
- 用户在地图任意位置"**留个味儿**"（发布文字留言，即"气味钉"）
- 留言附带坐标、时间戳、设备昵称 + 动物头像
- 其他用户进入该坐标 **50m 半径** 内可"**嗅一嗅**"（查看留言）
- **闻过**：用户查看气味钉时自动记录浏览次数，纯被动计数，无需任何操作
- **表态**：独立的主动互动，二选一——气味相投（支持）或臭味相投（不支持）；与闻过完全独立
- 支持回复（形成留言串）
- 支持**私密气味钉**：仅自己可见，永久保存，不过期，作为个人地图日记使用

**气味钉存活机制**（公开气味钉）
- 默认存活 **72小时**，到期气味消散
- **互动续命**：收到回复或表态均可延长存活时间（具体延长规则待定）
- **道具续命**：使用道具也视为一种互动，效果由道具类型决定（道具系统细节后续规划）
- 私密气味钉永久保存，不受此机制约束

**变现方向**
- 道具系统（付费道具 / 积分兑换）：细节后续规划
- 企业官方气味钉合作（见活动场景模式）

**边界 & 不做**
- 不做全量地图浏览（防止爬取、隐私问题）
- 不允许上传图片/视频（MVP 阶段）
- 单条留言字数上限 **280 字**（参考 Twitter 标准）
- 无需注册，设备首次使用自动生成身份

**活动场景模式**（演唱会 / 球赛等）
- **官方气味钉**：由主办方/合作企业发布，长期存在，不受活动时间限制
  - 活动合约履行完毕后自动**隐藏**（非删除）
  - 隐藏状态可通过新合约**重新激活**显示
- **公开个人气味钉**：活动场地范围内存活时间缩短为 **6小时**
  - 6小时到期后，内容自动**并入官方气味钉的留言区**保存，不丢失
  - 此后个人气味钉本身从地图上消失
- **私密气味钉**：永久遵循私密规则，不受活动场景约束，不限时、不并入官方钉

#### 模块 B：行进社交（Motion Chat）— **Phase 2，不在 MVP 范围**

**核心功能**
- 用户开启"行进模式"，每 10 秒上报一次 GPS 坐标
- 服务端通过**轨迹聚类算法**判断哪些用户处于同一交通工具上（地铁/高铁/公交/长途大巴等）
  - 判断条件：同一时间窗口内，速度 > 30km/h，多个用户坐标差 < 100m，且轨迹向量相似度高
- 系统识别到"同行者"后，向用户推送可邀请列表（展示匿名昵称 + 动物头像）
- **邀请制聊天室**：不自动拉人，由用户主动邀请或指名特定同行者
  - 支持 1 对 1 私聊，也支持 1 对多小组（上限 **10 人**）
  - 受邀者可接受或拒绝
- 行进结束（速度降至步行）或用户主动离开时，聊天室触发**结束流程**：
  - 选项 A：转为普通聊天室，双方可继续保持联系
  - 选项 B：原地解散——聊天记录**直接删除，不持久化**；平台不保留任何消息内容

**边界 & 不做**
- 不区分具体交通工具种类（统一速度阈值，覆盖所有移动场景）
- 不做实名验证
- 聊天室上限 **10 人**（强调小圈子亲密感，非广场式聊天）
- 解散后消息直接删除，服务端不持久化任何聊天内容

#### 模块 C：身份系统（无注册 + 封禁兜底）

- **设备绑定身份**：首次启动自动生成，无需邮箱/手机号
- 随机组合昵称（形容词 + 动物名，如"慵懒的水獭"）+ 对应动物头像
- 用户可手动改昵称和头像主题色，但无法彻底隐藏身份（防止高频恶意行为）
- 举报功能：每条气味钉均有举报按钮；累计举报次数达到阈值（默认 **3 次**，后续可调整）后自动隐藏内容，进入人工审核队列；审核通过后恢复显示，否则永久删除
- 屏蔽功能：屏蔽某只"动物"，再也闻不到 ta 的味儿
- 不强制实名，不公开地理位置历史

**身份恢复（换机迁移）**
- 首次启动时，客户端本地生成一组**私密恢复码**（12 个助记词），**Peemail 服务器不持有、不记录恢复码**
- 系统展示一次后由用户自行保管（截图、抄写等）；丢失后无法找回
- 换设备时输入恢复码，可完整恢复：昵称、动物头像、积分、道具
- **私密气味钉的加密密钥从恢复码派生**，无需单独备份，输入恢复码即可重新获得日记访问权
- 服务器只存储从主密钥派生的哈希值（用于恢复时校验身份），无法反推恢复码

**封禁机制（恶意用户兜底）— Phase 2**
- 设备 Token 被封禁后，**正常功能不可用**，进入手机号验证流程
- 完成手机号验证后方可重新进入（一次性解封）
- 手机号验证**仅在封禁场景触发**，正常使用无需注册，对绝大多数用户完全透明
- 同一手机号关联的历史设备记录保留——若同号再次被封，审核门槛提高，防止反复绕过

#### 模块 D：通信层

| 阶段 | 方式 | 用途 |
|------|------|------|
| MVP | WebSocket（服务器中转） | 留言实时推送（附近新钉、表态变化） |
| Phase 2 | WebSocket + GPS 聚类 + 服务器中转 | 行进模式同行者识别、邀请制聊天室；iOS ↔ Android 跨平台同行者发现均走此通道 |
| Phase 3 | WebSocket + BLE | 近场直连（断网或低信号场景） |

## 三、用户故事（核心场景）

### 场景 1：城市探索（MVP 核心场景）
> 小红路过一家咖啡店，打开 Peemail，地图上飘着几枚气味钉——得走进 50m 才能嗅到内容。"慵懒的水獭"留了味儿："这家拿铁很苦，别踩坑"——43 只动物闻过（自动计数），12 只表了态，8 只气味相投，4 只臭味相投。她在隔壁公园留个味儿，写了 280 字的碎碎念，顺手设为私密——这条日记永久保存，只有她自己能看到。

### 场景 2：演唱会现场
> 某乐队开演唱会，主办方设置了官方气味钉：入场须知、彩蛋提示、签售信息，长期存在。观众们在场内留下数百枚个人气味钉，系统自动设为 6 小时有效期——6 小时后这些内容并入官方气味钉的留言区保存，不丢失，但地图上不再显示。演出合约履行完毕后，官方气味钉自动隐藏，若主办方再次签约，原钉重新激活。

### 场景 3：早高峰通勤（Phase 2）
> 小明坐地铁，开启行进模式。系统发现附近有 5 名同行者，推送了可邀请列表。他点开"急躁的仓鼠"，发送邀请："同方向吗？"对方接受，两人开了个 1 对 1 小聊。到站下车，系统提示：选择转为普通聊天室继续联系，还是原地解散？他选择了保留。

### 5.2 技术选型详解

#### 前端

| 技术 | 版本建议 | 理由 |
|------|---------|------|
| **Expo (React Native)** | SDK 52+ | 一套代码 iOS + Android + Web；Expo Router 支持 SSR；生态最成熟 |
| **React Native Maps + Mapbox SDK** | latest | 封装 Mapbox，iOS/Android 原生地图；Web 端用 Mapbox GL JS |
| **Mapbox GL JS** | v3 | Web 端地图，矢量瓦片性能好，国内外统一服务商 |
| **Zustand** | v5 | 轻量状态管理，比 Redux 简洁 |
| **Socket.io Client** | v4 | 与后端 Socket.io 配套，自动重连 |

> **为什么不用 Flutter？** Dart 生态与 Web 互通成本高；React Native Web 能复用更多业务逻辑（尤其是地图和 WebSocket 代码）。

#### 后端

| 技术 | 理由 |
|------|------|
| **Node.js + Hono** | Hono 是轻量级框架，支持 Edge Runtime，类型安全；Node.js 生态对 WebSocket 友好 |
| **Socket.io** | 成熟的 WebSocket 方案，内置 room 管理（完美匹配"列车聊天室"模型），自动降级 |
| **PostgreSQL + PostGIS** | PostGIS 的 `ST_DWithin` 原生支持半径查询，地理围栏性能极好 |
| **Redis** | 在线用户状态、GPS 坐标缓存（TTL 30s）、聊天室成员列表 |
| **BullMQ** | GPS 聚类计算放入队列异步处理，避免阻塞主线程 |

> **为什么不用 Go？** 列车聚类逻辑计算复杂，Node.js 生态（lodash、turf.js）更丰富；如后期性能瓶颈明确，GPS Worker 可单独用 Go 重写。

#### 基础设施

| 服务 | 选型 | 备注 |
|------|------|------|
| 云平台 | AWS / 阿里云 | 国内用阿里云，出海用 AWS |
| 对象存储 | S3 / OSS | 图片存储 |
| CDN | Cloudflare | 静态资源加速 |
| 推送通知 | Expo Push + APNs/FCM | Expo 统一封装 |
| 监控 | Sentry + Grafana | 错误追踪 + 性能监控 |

## 六、核心算法说明

### 6.1 行进识别算法（Phase 2）

```
每 10s 收到用户 GPS 上报后：

1. 速度过滤：计算相邻两点速度，speed = distance / time
   - speed < 5 km/h → 步行/静止，不触发行进模式
   - 5 ≤ speed < 30 → 骑行/驾车，不触发
   - speed ≥ 30 → 进入候选（覆盖地铁/高铁/公交/大巴等）

2. 空间聚类：对所有"候选"用户，在 Redis 中做半径 100m 空间桶
   - 同一时间窗口内坐标距离 < 100m 的用户归为一组

3. 轨迹一致性验证：
   - 取过去 60s 的轨迹向量，计算组内用户向量余弦相似度
   - 相似度 > 0.95 → 确认同一交通工具

4. 推送可邀请列表：
   - 将同组用户列表（匿名身份）推送给各用户
   - 用户主动发出邀请，对方接受后通过 Socket.io join 同一 room
   - room 上限 10 人
```

### 6.2 地图留言查询

```sql
-- 查询用户 50m 范围内的有效留言（公开 + 本人私密）
-- 私密气味钉永久保存（expires_at IS NULL），公开气味钉有过期时间
-- is_hidden = true 的气味钉对所有人不可见（举报阈值触发，待审核）
SELECT id, content, created_at, lat, lng, is_private
FROM pins
WHERE ST_DWithin(
  location::geography,
  ST_MakePoint(:lng, :lat)::geography,
  50  -- 单位：米
)
AND is_hidden = false
AND (
  (is_private = false AND expires_at > NOW())   -- 公开且未过期
  OR (is_private = true AND owner_device_id = :device_id)  -- 私密且是本人
)
ORDER BY created_at DESC
LIMIT 50;
```

## 七、隐私与安全设计

| 风险 | 应对措施 |
|------|---------|
| 用户实时位置泄露 | GPS 坐标只存 Redis（TTL 30s），不写入持久化 DB |
| 留言内容违规 | 举报按钮（MVP）：累计达到阈值自动隐藏 + 人工审核队列；Phase 2 补充文字过滤 + 图片 AI 审核 |
| 行进聊天记录泄露 | 解散时消息直接删除，服务端不持久化任何聊天内容（Phase 2） |
| 用户身份追踪 | 设备绑定身份，不绑定真实信息；设备 ID 哈希后存储，无法反查 |
| 换机身份丢失 | 恢复码由客户端本地生成，服务端零知识；用户自行保管，丢失后无法找回（明确告知） |
| 频繁 GPS 上报耗电 | 用户可手动关闭行进模式（Phase 2）；后台时降低频率到 30s/次 |
| 恶意用户绕过封禁 | 封禁后强制手机号验证解封；同一手机号关联设备历史保留，多次违规提高封禁门槛（Phase 2） |

2. **道具系统定价**：积分获取速率 & 付费道具价格带（影响变现预期，后续专项规划）
3. **活动场景触发机制**：由运营手动配置 vs 通过 POI 数据自动识别演出场馆
4. **封禁门槛**：举报多少次触发封禁？是否分级（临时封禁 / 永久封禁）？
5. **手机号验证服务商**：国内用阿里云短信 / 腾讯云，海外用 Twilio，还是统一方案？
6. **BLE 方案选型**：React Native BLE PLX vs Expo BLE（Phase 3，后者尚不成熟）
7. **内容审核资源**：MVP 阶段是否引入第三方审核 API（如阿里云绿网）
8. **动物头像版权**：自绘插画 vs 购买授权素材库
9. **行进模式隐私提示**（Phase 2）：需设计明确的"我正在上报位置"UI 提示

## 十、开源策略与商业化

### 10.1 开源协议

**License：AGPL-3.0**

选择 AGPL-3.0 的原因：
- 代码**完全开源**，社区可自由查看、修改、部署
- 任何基于 Peemail 的网络服务修改后也必须回馈开源，防止商业闭源分叉
- 官方托管服务（peemail.app）为标准产品，用户可直接使用

### 10.2 社区贡献

- 欢迎社区提交 PR：功能改进、BUG 修复、语言包、地图皮肤等
- 贡献指南和 Code of Conduct 随代码仓库一同发布
- 核心功能决策由维护团队把关，UI/UX 实验性功能开放社区主导

### 10.3 商业化方向

| 方向 | 说明 |
|------|------|
| **付费道具 / 积分系统** | 延长气味钉存活、特效气味钉展示等；积分可通过日常互动获得，高级道具付费购买 |
| **企业官方气味钉合作** | 品牌 / 主办方签约后可发布官方气味钉，按合约期计费；合约到期后自动隐藏，续约重新激活 |
| **自托管友好** | 开源代码支持企业内网或社区自托管，官方不对自托管收费，但不提供官方支持 |
